Shape Rotator OS / transcript handling

Run one shared calendar, capture future transcripts, and send safe insights to the app.

The goal is operational first: the group gets a real calendar that is easy to update, meetings can create transcript sources automatically, and the app gets useful distilled context.

How it works: Calendar owns the event, Drive stores the raw transcript, AI drafts evidence, humans review it, and Supabase exports only approved insight records to the app-facing layer.

System flow calendar -> transcript -> review -> app

AI and security boundary

AI drafts. Supabase gates. Humans approve.

This is the rule that matters: AI can create drafts from private material, but it cannot publish, grant access, or decide what users can see.

What never crosses

App and public views omit raw transcript text, source storage paths, job IDs, private contacts, Google tokens, and service credentials.

If review is missing, a gate fails, or public text names people when it should not, the output stays held.

Never The browser receives service keys or Google tokens.
Never AI publishes public articles without review.
Never T3 articles name people by default.

Admin surface

What organizers need to operate.

This is the admin checklist. It keeps Calendar, Drive, Supabase, the app, and the site in sync.

event
Calendar event has a type, guests, and a Meet link.

The app can show subscribe and join links only when the event has the right calendar data.

source
Transcript lands in the right Drive folder.

Name sources type_project_name_date. Admins Tina, Andrew, Dmarz, Michael, Fred, and Albi need admin access to the managed transcript drives.

guest
Cohort guest selection uses a private directory.

The admin app should read private_invite_contacts. Public bundles should not contain the full contact list.

release
Updates push from reviewed Supabase rows.

The app and website should receive only approved distillations, evidence cards, and public-safe article exports.
