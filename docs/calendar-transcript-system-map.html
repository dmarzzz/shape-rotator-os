Shape Rotator OS / Meeting knowledge architecture

Meetings become trusted knowledge only after checks.

Google Calendar schedules the session. Google Meet runs it. Gemini transcript output and source files stay private. Supabase is the control plane: refs, jobs, evidence cards, review state, and approved distillations. The app reads approved views, not raw transcripts.

Tool chain

Google Calendar owns time, guests, session type, and Meet links.

Google Meet + Gemini produce the meeting transcript source.

Google Drive is the private source vault. Supabase stores metadata, jobs, private invite contacts, evidence cards, permissions, and reviewed outputs.

Security rule: browser clients never receive the raw transcript lane.

Organizer manual

How to update the system.

If organizers follow these four steps, the system stays reliable and avoids duplicate handling.

1. Update Calendar Create or edit the session in the managed Google Calendar or admin app. Do not create parallel invites. Timed sessions need a Meet link.
2. Store Source Name the transcript type_project-name_date. Put raw files in the correct private Drive folder, not in GitHub, public docs, or calendar descriptions.
3. Register Ref Supabase records the source as a ref with type, tier, storage mode, hash, and job state. The app does not need the raw file.
4. Review Output Review derived summaries and evidence cards. T2 can reach the cohort after review. T3 public needs all gates cleared.

Security model

Supabase records the path to insight, not the raw room.

Raw material stays in the private source lane. Supabase turns it into controlled rows, then exposes only approved views.

  • ingest

    poll-drive-artifacts finds Drive source refs; ingest-artifacts writes ingestion_events and source_artifacts: source type, tier, hash, size, uploader, and whether raw is server-readable.

  • queue

    processing_jobs says what must happen next: fetch an external ref, prepare review, or distill allowed text. Overlapping workers claim jobs atomically.

  • derive

    process-transcript-jobs runs server-side, fetches only allowed private sources, and writes derived_artifacts plus evidence_cards.

  • review

    review-transcript-artifact, artifact_reviews, and approval_gates record who approved, blocked, or held the output before it can move to the app or public site.

No service keys in the browser The web/app client uses browser-safe Supabase config. Google tokens and privileged Supabase credentials live in Edge Functions or trusted operator shells.
No raw transcript table for app users App users do not read source_artifacts, processing_jobs, or the private evidence_cards table. Those are coordinator/operator lanes behind RLS.
App views strip provenance internals app_transcript_distillations and app_transcript_evidence_cards expose approved content and omit raw source refs, processing job IDs, and private artifact rows.
T3 has a database stop sign Public output must be a public_candidate, approved, no-name for articles, and have all required gates cleared before publication.

Input hygiene

Every source is classified before it is trusted.

The system does not treat all transcripts as equal. It records where each source came from, how raw it is, who can inspect it, and whether it can be processed.

Meet / Gemini meet_transcript or meet_smart_notes. Stored in Drive, registered in Supabase, distilled only after the source ref exists.
Otter otter_transcript, otter_summary, or otter_slide. Slide/image artifacts stay private until a reviewed derivative is promoted.
Manual / raw logs manual_upload, local_only, or external_ref. Defaults to T0/T1 handling and waits for review before wider use.
Other signals GitHub, Router, notes, or docs can become evidence inputs. They enter as references and claims, not as noisy feeds pushed straight to the app.

Storage model

One meeting produces separate records.

The transcript source, evidence cards, reviewed app readout, and public article candidate are separate things.

Google Drive source

Stores Gemini transcript output, recordings, notes, or uploaded source files.

Drive folder permissions decide who can see raw material.

Supabase app record

Internally stores event metadata, source refs, tier, review state, derived readouts, evidence cards, actions, and provenance.

Users get calendar state and join event links when a Meet URL is available.

Website export

Contains only approved public material.

Article output is general and no-name by default unless a public gate explicitly allows more.

Admin path

Admins control routing before anyone reads output.

The admin view sets who attends, where raw files land, which tier applies, and what can be published.

  • type

    Name every transcript as type_project-name_date, then attach the session type and sharing ceiling.

  • route

    Google Drive sends raw files into tiered folders. T0/T1/T2/T3 decide who can see what.

  • sync

    Calendar changes flow into Supabase so the app sees current events, guests, Meet links, and review state.

  • hold

    Anything raw, ambiguous, private, or ungated stays out of app/public output.

Subscribe link Public users get read-only .ics / webcal subscription links.
Join link App users see join event when a Supabase event carries a Meet URL.
Guest picker Guest selection reads from private_invite_contacts, not the public bundle. Manual attendee paste is the fallback.
Raw transcript Never used as a direct app payload. It must become reviewed derived content first.

Failure behavior

If a step is missing, the system should stop narrow.

The safe failure mode is delay, not accidental publication.

No Google Meet link The event cannot produce the expected Meet/Gemini transcript path until repaired.
No Gemini transcript The meeting can still be tracked, but distillation waits for a Drive source file or manual upload.
No Supabase review state The output remains held. It does not become visible app or website content.
Public gate blocked No public article. T3 requires cleared gates.

Tool boundaries

No tool gets more data than it needs.

This is the main privacy rule in the architecture.

  • drive

    Google Drive can hold raw source material. Access is folder-based and narrow.

  • db

    Supabase holds metadata, source references, review state, and reviewed distillations.

  • app

    The app reads approved records, provenance, confidence, actions, and calendar state.

  • site

    The website only gets public-safe exports: general, reviewed, and no-name by default.

Short version: Google Calendar creates the session; Meet/Gemini creates the source; Drive stores the raw files; Supabase stores reviewed knowledge; the app and website read only the layer they are allowed to show.
