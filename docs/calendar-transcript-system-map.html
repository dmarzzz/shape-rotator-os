Shape Rotator OS / Meeting knowledge architecture

Meetings move through tools, then become reviewed knowledge.

Google Calendar schedules the session. Google Meet runs it. Gemini transcript output and source files stay private. Supabase is the control plane: refs, jobs, review state, and approved distillations. The app reads approved rows, not raw transcripts.

Tool chain

Google Calendar owns time, guests, session type, and Meet links.

Google Meet + Gemini produce the meeting transcript source.

Google Drive is the private source vault. Supabase stores metadata, jobs, permissions, and reviewed outputs.

Security rule: browser clients never receive the raw transcript lane.

The simple version

Each tool has one job.

The raw meeting never jumps straight into the app. It passes through capture, storage, processing, and review first.

1. Google Calendar Creates the event, guest list, session type, sharing ceiling, and Meet link. schedule
2. Meet + Gemini Runs the meeting and creates transcript output or capture artifacts for timed sessions. source
3. Google Drive Stores raw files in private folders. Folder permissions follow the session tier. vault
4. Supabase Records the event, source refs, access tier, processing job, and derived records. state
5. Review gates Admins approve summaries, evidence cards, actions, and public-safe outputs. review
6. App + site The app reads reviewed Supabase outputs. The website only gets cleared public material. surface

Security model

Supabase records the path to insight, not the raw room.

Raw material stays in the private source lane. Supabase turns it into controlled rows, then exposes only approved views.

  • ingest

    ingest-artifacts writes ingestion_events and source_artifacts: source type, tier, storage ref, hash, size, uploader, and whether raw is server-readable.

  • queue

    processing_jobs says what must happen next: fetch an external ref, prepare review, or distill allowed text. Overlapping workers claim jobs atomically.

  • derive

    process-transcript-jobs runs server-side, fetches only allowed private sources, and writes derived_artifacts: summaries, claims, Q&A, deltas, and public candidates.

  • review

    artifact_reviews and approval_gates record who approved, blocked, or held the output before it can move to the app or public site.

No service keys in the browser The web/app client uses browser-safe Supabase config. Google tokens and privileged Supabase credentials live in Edge Functions or trusted operator shells.
No raw transcript table for app users App users do not read source_artifacts or processing_jobs. Those are coordinator/operator lanes behind RLS.
App view strips provenance internals app_transcript_distillations exposes approved derived content and omits raw source refs, processing job IDs, and private artifact rows.
T3 has a database stop sign Public output must be a public_candidate, approved, and have all required gates cleared before publication.

Storage model

One meeting produces three records.

The transcript source, the reviewed app record, and the public article record are separate things.

Google Drive source

Stores Gemini transcript output, recordings, notes, or uploaded source files.

Drive folder permissions decide who can see raw material.

Supabase app record

Internally stores event metadata, source refs, tier, review state, claims, summaries, actions, and provenance.

Users get calendar state and join event links when a Meet URL is available.

Website export

Contains only approved public material.

Article output is general and no-name by default unless a public gate explicitly allows more.

Admin path

Admins control routing before anyone reads output.

The admin view sets who attends, where raw files land, which tier applies, and what can be published.

  • type

    Name every transcript as type_project-name_date, then attach the session type and sharing ceiling.

  • route

    Google Drive sends raw files into tiered folders. T0/T1/T2/T3 decide who can see what.

  • sync

    Calendar changes flow into Supabase so the app sees current events, guests, Meet links, and review state.

  • hold

    Anything raw, ambiguous, private, or ungated stays out of app/public output.

Subscribe link Public users get read-only .ics / webcal subscription links.
Join link App users see join event when a Supabase event carries a Meet URL.
Guest picker Guest selection should read from a private cohort contact source, not a public bundle. Manual attendee paste is the fallback.
Raw transcript Never used as a direct app payload. It must become reviewed derived content first.

Failure behavior

If a step is missing, the system should stop narrow.

The safe failure mode is delay, not accidental publication.

No Google Meet link The event cannot produce the expected Meet/Gemini transcript path until repaired.
No Gemini transcript The meeting can still be tracked, but distillation waits for a Drive source file or manual upload.
No Supabase review state The output remains held. It does not become visible app or website content.
Public gate blocked No public article. T3 requires cleared gates.

Tool boundaries

No tool gets more data than it needs.

This is the main privacy rule in the architecture.

  • drive

    Google Drive can hold raw source material. Access is folder-based and narrow.

  • db

    Supabase holds metadata, source references, review state, and reviewed distillations.

  • app

    The app reads approved records, provenance, confidence, actions, and calendar state.

  • site

    The website only gets public-safe exports: general, reviewed, and no-name by default.

Short version: Google Calendar creates the session; Meet/Gemini creates the source; Drive stores the raw files; Supabase stores reviewed knowledge; the app and website read only the layer they are allowed to show.
